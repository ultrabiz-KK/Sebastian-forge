### Task A: Settings.tsx のマスターパスワード関連修正

#### A-1: マスターパスワードのスイッチデザインを他のスイッチと統一する

**タイトル:** 設定画面のマスターパスワードスイッチのデザインを他のトグルスイッチと統一する

**説明:**

1. **使用技術**

2. **具体的な作業内容**

   - 他のスイッチ: `<div className="flex items-center justify-between">` で直接ラベルとトグルを配置
   - マスターパスワード: `bg-sebastian-parchment/50` コンテナ内に入れ子になっており、Shieldアイコン＋バッジ付きで複雑な構造
   - トグルのON/OFF動作（モーダル表示・確認ダイアログ）は維持しつつ、見た目を統一する

3. **対象ファイル・関連箇所**

4. **完了条件**

   - ON/OFF時の動作（モーダル・確認ダイアログ）は変更なし

5. **制約・やってはいけないこと**
   - トグルのクリック動作ロジックを変更しない
   - 他のセクションのスイッチデザインを変更しない

#### A-2: Settings初期ロードでAPIキーを復号して読み込む（マスターパスワード有効時のAPI接続修復）

**タイトル:** Settings.tsx のAPIキー読み込みを `getDecryptedSetting()` に変更し、二重暗号化を防止する

**説明:**

1. **使用技術**
   - TypeScript
   - `settings.ts` の `getDecryptedSetting()`

2. **具体的な作業内容**
   - **根本原因**: Settings.tsx の初期ロード（`load()` 関数）で全APIキーを `getSetting()` で読み込んでいるため、暗号化済みの値 `ENC:...` がそのままフォームステートに格納される
   - これにより:
     (a) フォームに暗号化文字列が表示される
     (b) 再保存時に `setEncryptedSetting()` が暗号化文字列をさらに暗号化（二重暗号化）
     (c) 二重暗号化された値は復号できず、API接続が失敗する
   - **修正**: APIキーの読み込みを `getSetting()` → `getDecryptedSetting()` に変更する
   - セッション未解除時は `getDecryptedSetting()` が `null` を返すので、フォームには空文字を表示する（既存APIキーが消えてしまうように見えるが、保存しなければ上書きされない）
   - 保存時に空のAPIキーを暗号化保存しないようガードを追加する

3. **対象ファイル・関連箇所**
   - `src/pages/Settings.tsx` — `load()` 関数（L198〜L315）、`handleSave()` 関数（L440〜L509）

4. **完了条件**
   - マスターパスワード有効＋セッション解除状態で、APIキーが正しく復号されてフォームに表示される
   - 設定保存後もAPIキーが正しく暗号化され、AI機能が動作する
   - 二重暗号化が発生しない

5. **制約・やってはいけないこと**
   - マスターパスワード未設定時の既存動作（平文保存）を壊さない
   - セッション未解除時にフォームの空値を上書き保存しない（ユーザーが明示的に変更した場合のみ保存）

#### A-3: APIキー入力後にフィールドをグレーアウトし、編集ボタンで編集可能にする

**タイトル:** APIキー入力フィールドを保存後に読み取り専用にし、編集ボタンで編集を許可する

**説明:**

1. **使用技術**
   - React, TailwindCSS v4

2. **具体的な作業内容**
   - `ApiKeyField` コンポーネントを拡張:
     (a) APIキーが入力済み（空でない）の場合、テキストフィールドをグレーアウト（`readOnly` + 背景色変更）
     (b) 「編集」ボタン（Pencilアイコン）を追加。クリックで編集モードに切り替え
     (c) マスターパスワード有効中に編集ボタンを押した場合:
       - テキストフィールドの値をクリア（暗号化されたキーを表示しない）
       - 空の状態から新しいキーを入力できるようにする
     (d) 編集モード中は「キャンセル」で元の値に戻せる

3. **対象ファイル・関連箇所**
   - `src/pages/Settings.tsx` — `ApiKeyField` コンポーネント（L1438〜L1475）
   - `src/pages/Settings.tsx` — 各プロバイダー設定での `ApiKeyField` 呼び出し箇所

4. **完了条件**
   - 保存済みAPIキーがある場合、フィールドがグレーアウトされて編集不可
   - 編集ボタンをクリックすると編集可能になる
   - マスターパスワード有効時、編集開始時にフィールドがクリアされる
   - キャンセルで元に戻せる

5. **制約・やってはいけないこと**
   - 既存の表示/非表示トグル（Eye/EyeOff）機能を削除しない
   - 初回入力時（値が空の場合）はグレーアウトしない

### Task B: マスターパスワード設定後の自動セッション有効化

**タイトル:** マスターパスワード初回設定後に自動的にセッションを有効化する

**説明:**

1. **使用技術**
   - TypeScript
   - `session.ts` の `unlock()` 関数

2. **具体的な作業内容**
   - **根本原因**: `MasterPasswordSetupModal.tsx` の `handleSetup()` でパスワードハッシュをDBに保存した後、`session.ts` の `unlock()` を呼んでいない
   - そのため、パスワード設定直後にセッションが無効のまま → APIキーの暗号化も復号もできない → API接続不可
   - **修正**:
     (a) `handleSetup()` の成功処理内で `unlock(newPassword)` を呼び出す
     (b) `handleChangeConfirm()` でも同様にパスワード変更後に `unlock(newPassword)` を呼ぶ
     (c) Settings.tsx 側の `onPasswordSet` コールバックで `setSessionState(getState())` を呼び直す（既存）

3. **対象ファイル・関連箇所**
   - `src/components/MasterPasswordSetupModal.tsx` — `handleSetup()`（L37〜L64）, `handleChangeConfirm()`（L90〜L117）
   - `src/lib/session.ts` — `unlock()` のインポート確認

4. **完了条件**
   - マスターパスワードを初回設定した直後にセッションが有効状態になる
   - Settings画面のセッション状態表示が「セッション有効」に即座に更新される
   - パスワード設定後すぐにAPIキーの暗号化保存が機能する

5. **制約・やってはいけないこと**
   - `unlock()` の既存ロジック（ハッシュ検証 → セッション開始）を変更しない
   - パスワード削除（`handleDelete`）時にはセッション有効化しない
