## 概要

Phase 2 完了後のコード精査で発見された高重要度の問題6件を、Phase 3 着手前に修正する。

## 技術的アプローチ

### SR-1: パストラバーサル修正
- `validate_path()` ヘルパー関数を `lib.rs` に追加
- `canonicalize()` でパスを正規化し、許可ディレクトリリストと照合
- 許可ディレクトリ: アプリデータディレクトリ（`app_data_dir()`）、ユーザー設定の保存先パス

### SR-2: withModelOverride実装
- 各プロバイダークラスにファクトリパターンでクローンメソッドを実装
- `_modelOverride` プロパティは既に各クラスに存在するため、コピーロジックのみ追加
- `as any` キャストを除去して型安全性を回復

### SR-3: 暗号化フォールバック除去
- `setEncryptedSetting()` のcatchブロックを修正: 平文保存ではなくthrow
- マスターパスワード未設定時の後方互換を `isMasterPasswordSet()` で判定

### SR-4: セッションクリア
- `handleDelete()` に `lock()` 呼び出しを1行追加するだけの最小変更

### SR-5: カスタムプロバイダー復号
- `getDecryptedSetting()` で `ENC:` プレフィックスの有無で復号を判定する方式に変更
- `ENCRYPTED_KEYS` は保存時の暗号化判定にのみ使用

### SR-6: エラー握りつぶし修正
- 全catchブロックに `console.error()` または `console.warn()` を追加
- API呼び出し系は `error.message` を含む詳細ログ
- モデルリスト取得系は `console.warn`（ユーザー影響が小さいため）

## ビルド検証

| SR-2〜SR-6 | `npm run build`（`tsc && vite build`） |

## リスク

| リスク | 対策 |
|--------|------|
| SR-1 でユーザー設定パスのファイル書き出しが壊れる | 日報/週報パスを許可リストに含める |
| SR-3 でマスターパスワード未設定ユーザーが保存不可になる | `isMasterPasswordSet()` で分岐 |
| SR-5 で既存の暗号化済みカスタムキーが復号不可になる | `ENC:` プレフィックス方式なら後方互換 |
